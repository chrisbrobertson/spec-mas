## Feature 1: Architecture Analysis & Spec Splitting

### What It Does

Automatically detects when specifications are becoming too large or covering multiple concerns, and provides intelligent recommendations for splitting them into smaller, more focused specifications.

### Why It Matters

- **Reduces Implementation Risk**: Smaller specs are less likely to fail
- **Enables Parallel Development**: Split specs can be worked on simultaneously
- **Improves Quality**: Focused specs are easier to review and test
- **Faster Time to Value**: Can ship incremental features rather than waiting for monolith
- **Better Cost Management**: Smaller specs have more predictable API costs

### How It Works

#### Automatic Monitoring
Throughout the spec creation conversation, Claude now monitors:
- Number of business domains mentioned
- Count of functional requirements
- Data entity complexity
- API endpoint count
- External integrations
- User personas and workflows

#### Detection Triggers
Claude will alert you when:
- **Immediate**: You describe 3+ distinct domains in initial description
- **Progressive**: Functional requirements exceed 12, or other indicators accumulate
- **At validation**: Always checks before marking spec complete

#### Analysis Criteria

**Complexity Indicators:**
- Multiple user personas (3+)
- Many functional requirements (12+)
- Many data entities (8+)
- Many API endpoints (10+)
- Multiple external integrations (3+)
- Multiple workflows (4+)

**Cohesion Indicators:**
- Multiple business domains detected
- Different data lifecycle operations
- Different integration patterns

**Size Metrics:**
- Lines > 700 (warning), > 1000 (critical)
- Words > 3500 (warning), > 5000 (critical)

#### Split Recommendations

When splitting is recommended, Claude provides:
1. **Confidence Level**: How certain the recommendation is
2. **Specific Reasons**: Why splitting is beneficial
3. **Suggested Split**: Concrete proposal for how to divide the spec
4. **Priority**: Which parts to implement first
5. **Dependencies**: How split specs relate to each other

### Best Practices

**When to Split:**
- Total analysis score >= 10
- Claude recommends it during conversation
- Spec exceeds 700 lines
- Implementation estimate > 2 weeks
- Multiple distinct business domains

**How to Split:**
1. Identify natural boundaries (domains, workflows, integrations)
2. Ensure each new spec has single, clear purpose
3. Make each spec independently implementable
4. Document dependencies between split specs
5. Add cross-references in YAML front-matter
6. Validate each split spec individually

**Split Granularity:**
- Target: 3-8 FRs per spec
- Target: 400-600 lines per spec
- Target: 1-2 week implementation per spec
- Avoid: Specs with >3 dependencies on other specs

## Feature 2: Integrated Syntax Validation

### What It Does

Validates that specifications are syntactically correct and complete before marking them as "agent-ready". Catches common issues like missing sections, incomplete requirements, vague criteria, and placeholder text.

### Why It Matters

- **Prevents Implementation Failures**: Agents can't implement incomplete specs
- **Saves Time**: Catches issues before expensive AI agent execution
- **Ensures Consistency**: All specs meet minimum quality standards
- **Reduces Clarification Loops**: Complete specs = no questions during implementation
- **Improves Success Rate**: Validated specs have much higher implementation success

### How It Works

#### Validation Gates

**G1: Structure & Format**
- YAML front-matter present and valid
- Required metadata fields present
- Section structure matches template

**G2: Content Completeness**
- All required sections populated
- No placeholder text ([TBD], [TODO], etc.)
- Functional requirements have descriptions
- Data model has entities defined

**G3: Testability**
- Functional requirements have validation criteria
- At least 3 deterministic tests defined
- Acceptance criteria present
- Tests have concrete inputs/outputs

**G4: Implementation Readiness**
- Requirements are specific and measurable
- Security section complete
- API contracts defined (if applicable)
- No vague terms ("fast", "user-friendly", etc.)

#### Validation Timing

**During Development (Advisory):**
- Light checks at level transitions
- Warns about missing items
- Non-blocking

**Before Completion (Mandatory):**
- Full validation before marking "complete"
- Must pass to export spec
- Blocking

**User-Requested:**
- Via `/validate` command anytime
- Provides detailed report
- Suggests fixes

## Support & Troubleshooting

### Common Issues

**"Analysis says to split but I want one spec"**
- You can override the recommendation, but be aware of increased risk
- Consider implementing as phases within one spec instead
- Ensure you have adequate time and resources

**"Validation keeps failing"**
- Review the specific error messages
- Reference the validation-integration-guide.md
- Ask Claude: "/validate --fix" for guided remediation

**"Commands not working in Claude"**
- Verify system-prompt.md is correctly pasted
- Ensure knowledge files are uploaded
- Try restarting the conversation

- Check the guides: `architecture-analysis-guide.md`, `validation-integration-guide.md`
- Review examples in `specs/examples/`
- Ask Claude: "/help" for available commands
- Open GitHub issue for bugs or feature requests
